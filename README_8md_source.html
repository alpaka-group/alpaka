1 **alpaka** - Abstraction Library for Parallel Kernel Acceleration
2 =================================================================
3 
4 The **alpaka** library is a header-only C++11 abstraction library for accelerator development.
5 
6 Its aim is to provide performance portability through the abstraction (not hiding!) of the underlying levels of parallelism.
7 
8 It is platform independent and supports the concurrent and cooperative use of multiple devices such as CPUs, CUDA GPUs and Xeon Phis with a multitude of accelerator-backends such as CUDA, OpenMP (2.0), Boost.Fiber, std::thread and serial execution (where applicaple). Only one implementation of the kernel is required by utilizing the uniform kernel interface.
9 
10 The **alpaka** API is currently unstable (alpha state).
11 
12 The library allows users to utilize a multitude of different accelerator types that require different libraries/compilers by providing a uniform kernel interface.
13 Users have to write only one implementation of their algorithms and can benefit from all supported accelerators.
14 There is no need to write special CUDA, OpenMP or custom threading code.
15 The supported accelerators can be selected at compile time but the decision which accelerator executes which kernel can be made at runtime.
16 
17 The abstraction used is very similar to the CUDA grid-blocks-threads division strategy.
18 Algorithms that should be parallelized have to be divided into a 1, 2, or 3-dimensional grid consisting of small uniform work items.
19 The function being executed by each of this threads is called a kernel.
20 The threads in the grid are organized in blocks.
21 All threads in a block are executed in parallel and can interact via fast shared memory.
22 Blocks are executed independently and can not interact in any way.
23 The block execution order is unspecified and depends on the accelerator in use.
24 By using this abstraction the execution can be optimally adapted to the available accelerators.

70 Dependencies
71 ------------
72 
73 [Boost](http://boost.org/) 1.56+ is the only mandatory external dependency.
74 The **alpaka** library itself just requires header-only libraries.
75 However some of the examples require different boost libraries to be built.
76 
77 When the *CUDA accelerator back-end* is enabled, version *7.0* of the *CUDA SDK* is the minimum requirement.
78 
79 When the *OpenMP 2.0 accelerator back-end* is enabled, the compiler and the platform have to support *OpenMP 2.0* or newer.
80 
81 When the *OpenMP 4.0 accelerator back-end* is enabled, the compiler and the platform have to support *OpenMP 4.0* or newer.
82 
83 When the *Boost.Fiber accelerator back-end* is enabled, the develop branch of boost and the proposed boost library [`boost-fibers`](https://github.com/olk/boost-fiber) (develop branch) are required. `boost-fibers`, `boost-context` and all of its dependencies are required to be build in C++14 mode `./b2 cxxflags="-std=c++14"`.

86 Usage
87 -----
88 
89 The library is header only so nothing has to be build.
90 Only the include path (`-I` or `export CPLUS_INCLUDE_PATH=`) has to be set to `<PATH-TO-ALPAKA-LIB>/include/`.
91 This allows to include the whole alpaka library with: `#include <alpaka/alpaka.hpp>`
92 
93 Code not intended to be utilized by users is hidden in the `detail` namespace.
94 
95 If you are building with the *CUDA accelerator back-end* enabled, your source files are required to have the ending `.cu` to comply with the nvcc (NVIDIA CUDA C++ compiler) rules for code files using CUDA.
96 When the *CUDA accelerator back-end* is disabled, this is not required and a `.cpp` extension is enough.
97 To allow both use-cases, it is desirable to have both, a `.cpp` file with the implementation and a `.cu` file containing only `#include <PATH/TO/IMPL.cpp>` to forward to the implementation.
98 The build system then has to use the `.cu` files when the *CUDA accelerator back-end* is enabled and the `.cpp` files else.
