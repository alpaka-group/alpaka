1 Abstraction
2 ===========
3 
4 Parallelism and memory hierarchies at all levels need to be exploited in order to achieve performance portability across various types of accelerators.
5 
6 On the hardware side we have nodes with multiple sockets/processors extended by accelerators like GPUs or Intel Xeon Phi each with their own processing units.
7 Within a CPU or a Intel Xeon Phi there are cores with hyper-threads and vector units, within a GPU there are many small cores.
8 Each of those entities has access to different memories in the hierarchy.
9 For example each socket/processor manages its RAM and the cores additionally have access to L3, L2 and L1 caches.
10 On a GPU there is global, constant and shared memory.
11 **alpaka** is designed to abstract from these differences without sacrificing speed by defining a domain decomposition for the computation domain.
12 This domain decomposition is abstract enough to map optimally to all (currently known) accelerators.
13 The **alpaka** library hides the mapping to the underlying hardware together with the execution primitives like kernel-threads and fibers allowing accelerator independent performance and portability.
14 
15 A process running on a multi-socket node is the largest entity within **alpaka**.
16 The library itself only abstracts the task and data parallel execution on the process/node level and down.
17 It does not provide any primitives for inter-node communication but such libraries can build upon **alpaka**.
18 The process always has a main thread and is by definition running on the host.
19 It can access the host memory and various accelerator devices.
20 Such accelerator devices can be GPUs, Intel Xeon Phi, the host itself or other hardware.
21 Thus the host not necessarily has to be different from the accelerator device used to compute on.
22 For example a Intel Xeon Phi simultaneously can be the host and the accelerator device.
23 **alpaka** also allows the parallel execution on nodes without any accelerator hardware.
24 The node the process is running on can itself be used as an accelerator device.
25 
26 **alpaka** can be used to offload the parallel execution of task and data parallel work simultaneously onto different accelerator devices.

29 Task Parallelism
30 ----------------
31 
32 With the concept of streams known from CUDA, where each stream is a queue of sequential tasks, but streams can be processed in parallel, **alpaka** provides an implementation of task parallelism.
33 Events that can be enqueued to the streams enhance this basic task parallelism by allowing synchronization between different streams, devices or host threads.

36 Data Parallelism
37 ----------------
38 
39 The main part of **alpaka** is the way it abstracts data parallelism.
40 The work is divided into a 1 to 3 dimensional grid of uniform threads appropriate to the problem at hand.
41 The uniform function executed by each of the threads is called a kernel.
42 
43 The threads are organized hierarchically and can access local memory at each of the hierarchy levels.
44 All these higher levels are hidden in the internals of the accelerator implementations and their execution order can not be controlled.
45 
46 The abstraction used extends the CUDA grid-blocks-threads division strategy explained below by further allowing to facilitate vectorization.
47 This extended *redundant hierarchical parallelism* scheme is discussed in the paper [The Future of Accelerator Programming: Abstraction, Performance or Can We Have Both?](http://dx.doi.org/10.1109/ICPADS.2013.76) ([PDF](http://olab.is.s.u-tokyo.ac.jp/~kamil.rocki/rocki_burtscher_sac14.pdf)).
48 
49 ### Thread Hierarchy
50 
51 #### Grid
52 
53 The whole grid consists of uniform threads each executing the same kernel.
54 By default threads do not have a cheap way to communicate safely within the whole grid.
55 This forces decoupling of threads and avoids global interaction and global dependencies.
56 This independence allows scattering of work blocks within the grid of worker threads and their independent execution on separate processing units.
57 Utilizing this property on the higher level allows applications to scale very well.
58 All threads within the grid can access a global memory.
59 
60 #### Block
61 
62 A block is a group of threads.
63 The whole grid is subdivided into equal sized blocks.
64 Threads within a block can synchronize and have a fast but small shared memory.
65 This allows for fast interaction on a local scale.

71 Each thread executes the same kernel.
72 The only difference is the index into the grid which allows each thread to compute a different part of the solution.

76 #### Vectorization (SIMD)
77 
78 To use the maximum available computing power of a x86 core the computation has to exploit the vector registers.
79 
80 Because of the x86 SIMD intrinsics (`<xmmintrin.h>`) not being portable, we have to rely on the loop vectorization capabilities of the compiler.
81 
82 The best solution to vectorization would be one, where the user does not have to do anything.
83 This is not possible because the smallest unit the user supplies is a kernel which is executed in threads which can synchronize.
84 It is not possible to hide the vectorization by starting a kernel-thread for e.g. each 4th thread in a block and then looping over the 4 entries.
85 This would prohibit the synchronization between these threads.
86 By executing 4 fibers inside such a vectorization kernel-thread we would allow synchronization again but prevent the loop vectorizer from working.
87 
88 The only possible usage of vectorization is one where we create a kernel-thread for e.g. each 4th thread in a block but do not loop over the 4 threads ourself but rely on the user to implement loops that can be vectorized safely.

90 ### Memory Hierarchy
91 
92 #### Global Memory
93 
94 The global memory can be accessed from every thread executing on an accelerator.
95 This is typically the largest but also the slowest memory available.
96 
97 #### Shared Memory
98 
99 Each block has its own shared memory.
100 This memory can only be accessed by threads within the same block and gets discarded after the complete block finished its calculation.
101 This memory is typically very fast but also very small.
102 Sharing has to be done explicitly.
103 No variables are shared between kernels by default.
104 
105 #### Registers
106 
107 This memory is local to each thread.
108 All variables with default scope defined inside a kernel are automatically saved in registers and not shared automatically.

112 Mapping *Redundant Hierarchical Parallelism* onto Hardware
113 --------------------------------------------------------
114 
115 By providing an accelerator independent interface for kernels, their execution and memory access at different hierarchy levels **alpaka** allows the user to write accelerator independent code that does not neglect performance.
116 
117 The hard part, the mapping of the decomposition to the execution environment is handled by the **alpaka** library.
118 The decomposition of the computation in use can not be mapped one to one to any existing hardware.
119 GPUs do not have vector registers where multiple values of type `int` or `float` can be manipulated by one instruction.
120 Newer versions of CUDA only implement basic SIMD instructions "on pairs of 16-bit values and quads of 8-bit values".
121 They are described in the documentation of the [NVIDIA GPU instruction set architecture](http://docs.nvidia.com/cuda/pdf/ptx_isa_3.2.pdf) chapter 8.7.13 but are only of any use in very special problem domains.
122 So the vector level is omitted on the CUDA accelerator.
123 CPUs in turn are not (currently) capable of running thousands of threads concurrently.
124 Furthermore CPUs do not have an equivalently fast inter-thread synchronization and shared memory access as GPUs do.
125 
126 A major point of the *redundant hierarchical parallelism* abstraction is to ignore specific unsupported levels and utilize only the ones supported on a specific accelerator.
127 Furthermore the hierarchy allows a mapping to various current and future accelerators in a variety of ways enabling optimal usage of the underlying compute and memory capabilities.
